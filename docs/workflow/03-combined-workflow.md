# Combined Workflow

This is the **recommended setup** when a project uses both layers:

- **Shared skills** pulled from a central repo (see
  [`01-shared-skills.md`](./01-shared-skills.md)).
- **Project-only skills** authored in-tree (see
  [`02-project-skills.md`](./02-project-skills.md)).

The two layers coexist cleanly because the CLI's destructive operations are
keyed by **skill name**, not by directory. Only `.agents/skills/<name>/`
for the specific skill being installed gets wiped on reinstall; everything
else in `.agents/skills/` is untouched.

---

## Directory layout (target state)

```
my-service/
├── _skills/                            ← 🔒 source of truth for project-only (committed)
│   ├── migration-helper/
│   │   └── SKILL.md
│   └── oncall-runbook/
│       └── SKILL.md
│
├── .agents/                            ← 🗑 generated (gitignored)
│   └── skills/
│       ├── code-review/                ← shared (from acme/skills@v1.4.0)
│       ├── deploy-checklist/           ← shared
│       ├── migration-helper/           ← project-only (from ./_skills)
│       └── oncall-runbook/             ← project-only
│
├── <agent>/                            ← 🗑 generated (gitignored); one per targeted agent
│   └── skills/
│       ├── code-review/         → ../../.agents/skills/code-review/
│       ├── deploy-checklist/    → ../../.agents/skills/deploy-checklist/
│       ├── migration-helper/    → ../../.agents/skills/migration-helper/
│       └── oncall-runbook/      → ../../.agents/skills/oncall-runbook/
│
└── skills-lock.json                    ← 🔒 committed; both layers in one file
```

Concrete `<agent>` directories are `.claude/`, `.cursor/`, `.codex/`, etc.
A project targeting two agents has two such directories, each containing
symlinks into `.agents/skills/`.

Example `skills-lock.json` with both layers present:

```json
{
  "version": 1,
  "skills": {
    "code-review": {
      "source": "acme/skills",
      "ref": "v1.4.0",
      "sourceType": "github",
      "computedHash": "sha256:abc..."
    },
    "deploy-checklist": {
      "source": "acme/skills",
      "ref": "v1.4.0",
      "sourceType": "github",
      "computedHash": "sha256:def..."
    },
    "migration-helper": {
      "source": "./_skills",
      "sourceType": "local",
      "computedHash": "sha256:123..."
    },
    "oncall-runbook": {
      "source": "./_skills",
      "sourceType": "local",
      "computedHash": "sha256:456..."
    }
  }
}
```

`sourceType: "github"` entries are updated by `skills update -p`.
`sourceType: "local"` entries are skipped on update — they're managed via
the local sync script instead.

## The three core command sequences

Three operations cover the project-level lifecycle. They're plain CLI
invocations — wire them into whatever task runner your project uses
(`package.json` scripts, a `Makefile`, a `justfile`, shell aliases, etc.).
The docs below call them by name, but don't prescribe an implementation.

### Bootstrap

Run after `git clone` or whenever `.agents/` has been blown away.
Refreshes shared skills from their pinned refs, then re-syncs project-only
skills. Idempotent.

```bash
npx skills update -p -y
npx skills add ./_skills --skill '*' -a <agent> -y
```

Pass one `-a <agent>` per agent you target (e.g. `-a claude-code -a cursor`).

### Sync local

Run after editing any `_skills/<name>/SKILL.md` to push the change into
`.agents/skills/<name>/` and refresh symlinks. Skips shared skills entirely.

```bash
npx skills add ./_skills --skill '*' -a <agent> -y
```

### Update shared

Run to refresh shared skills. Bumps `computedHash` if any shared source
content changed; leaves local entries alone.

```bash
npx skills update -p -y
```

### Wiring into a task runner (optional)

Pick whichever fits your project. A few examples:

- **`package.json` scripts** — `"skills:bootstrap": "npx skills update -p -y && npx skills add ./_skills --skill '*' -a <agent> -y"`
- **`Makefile` target** — a `skills-bootstrap:` rule that runs the two commands.
- **`justfile` recipe** — `bootstrap-skills:` with each command on a new line.
- **Shell alias** — a function in `.zshrc` / `.bashrc` local to the repo.

The docs below refer to these by intent ("run the bootstrap commands", "run
the local-sync command") — substitute whatever your team has wired up.

### Onboarding a new teammate

```bash
git clone <repo>
cd <repo>
# Run the bootstrap commands (above)
```

That's it. Both layers are live for whichever agents the team targets.

## Day-to-day operations

### Adding a shared skill

```bash
npx skills add acme/skills#v1.4.0 --skill code-review
git add skills-lock.json
git commit -m "Add code-review skill from acme/skills@v1.4.0"
```

### Adding a project-only skill

```bash
mkdir -p _skills/migration-helper
cd _skills/migration-helper && npx skills init && cd -
# edit _skills/migration-helper/SKILL.md
npx skills add ./_skills --skill '*' -a <agent> -y   # sync local
git add _skills/migration-helper skills-lock.json
git commit -m "Add project-local migration-helper skill"
```

### Editing a project-only skill

```bash
# edit _skills/<name>/SKILL.md
npx skills add ./_skills --skill '*' -a <agent> -y   # sync local
git add _skills skills-lock.json
git commit -m "Refine migration-helper skill"
```

Only the `computedHash` in `skills-lock.json` changes for that skill — a
one-line lockfile diff.

### Updating a shared skill

Option 1 — bump a pin:

```bash
npx skills add acme/skills#v1.5.0 --skill code-review
git add skills-lock.json
git commit -m "Bump code-review to v1.5.0"
```

Option 2 — refresh against the current ref (if tracking a branch):

```bash
npx skills update -p -y
git add skills-lock.json
git commit -m "Refresh shared skills"
```

### Removing a skill (either layer)

```bash
npx skills remove <name>
# for a project-only skill, also remove the source:
rm -rf _skills/<name>
git add -A
git commit -m "Remove <name> skill"
```

`skills remove` works identically for both layers — deletes canonical copy,
symlinks, and lockfile entry.

### Promoting a project-only skill to the shared repo

```bash
# 1. In the shared repo
cp -r my-service/_skills/migration-helper acme-skills/skills/
cd acme-skills && git add skills/migration-helper && git commit && git tag v1.5.0 && git push --tags

# 2. In the project — flip layers
rm -rf _skills/migration-helper
npx skills remove migration-helper            # clears local lock entry
npx skills add acme/skills#v1.5.0 --skill migration-helper
git add -A
git commit -m "Promote migration-helper to acme/skills@v1.5.0"
```

The skill's **name stays the same** — consumers see zero disruption. Only
the lockfile entry's `sourceType` flips from `local` to `github`.

### Pruning orphaned skills when bumping a pin

When a shared skill is **removed upstream** at the new pin, the CLI doesn't
automatically drop it from your project. The lockfile entry stays, the
canonical copy stays, and any subsequent bootstrap reinstalls it from the
old pin — silently.

#### Two kinds of orphan

| Type | Example | Caught by |
|---|---|---|
| **Filesystem orphan** | Skill on disk but not in lockfile | Nuking `.agents/skills/` + re-bootstrap |
| **Lockfile orphan** | Entry in lockfile you no longer want | Requires explicit `skills remove` |

A filesystem-only nuke (`rm -rf .agents/skills <agent>/skills`) followed by
a rebootstrap cleans up stray files but **reinstalls every lockfile entry**
from the authoritative lockfile. That's correct behavior — but it means the
lockfile is the real source of truth, and any stale entry in it will keep
coming back.

#### Safe procedure: explicit prune on pin-bump

Treat a pin bump as a remove-then-add operation:

```bash
# 1. See what's installed from this shared source
npx skills list

# 2. Remove every skill currently sourced from the old pin
npx skills remove code-review deploy-checklist old-skill

# 3. Re-add only the skills that still exist at the new pin
npx skills add acme/skills#v1.4.0 --skill code-review --skill deploy-checklist

# 4. Resync local skills (not affected by steps 1-3, but good hygiene)
npx skills add ./_skills --skill '*' -a <agent> -y

# 5. Commit — the lockfile diff will show old-skill being dropped
git add skills-lock.json
git commit -m "Bump acme/skills to v1.4.0 (drop old-skill)"
```

The lockfile diff is the review artifact — reviewers see exactly which
skills were added, updated, and removed in one atomic commit.

#### `_skills/` orphans

If you delete a project-only skill's directory but forget to remove its
lockfile entry, the next local-sync will fail loudly (it can't find the
source in `_skills/`). Always pair the two:

```bash
rm -rf _skills/migration-helper
npx skills remove migration-helper
git add -A
git commit -m "Remove migration-helper skill"
```

#### Nuclear option: full reset

When a project has drifted badly (multiple pins mixed, stale entries,
unclear state), a full reset is faster than reconciling:

```bash
# 1. Back up the current lockfile for comparison
cp skills-lock.json skills-lock.json.bak

# 2. Nuke everything the CLI manages (repeat the <agent>/skills line per agent)
rm -rf .agents/skills <agent>/skills skills-lock.json

# 3. Re-add each shared skill explicitly at the intended pin
npx skills add acme/skills#v1.4.0 --skill code-review
npx skills add acme/skills#v1.4.0 --skill deploy-checklist

# 4. Re-add project-only skills
npx skills add ./_skills --skill '*' -a <agent> -y

# 5. Diff against the backup to confirm the intended changes
diff <(jq -S . skills-lock.json.bak) <(jq -S . skills-lock.json)
rm skills-lock.json.bak
```

This rebuilds the lockfile from scratch, guaranteeing no stale entries.
Use it when the lockfile has drifted enough that you no longer trust it.

#### Automating prune with an intent file (optional)

Teams that bump pins frequently can treat the lockfile as a **derived
artifact** by keeping a separate `skills.intent.json` at the project root:

```json
{
  "shared": {
    "acme/skills": {
      "ref": "v1.4.0",
      "skills": ["code-review", "deploy-checklist"]
    }
  }
}
```

A reset script reads the intent file, removes any lockfile entry not listed
there (or not found in `_skills/`), then rebuilds. With this in place,
bumping a pin becomes a single edit to `skills.intent.json` followed by
running the reset, and CI can fail if the lockfile drifts from the intent.

Worth the complexity only if:

- Pin bumps happen often enough to be painful manually.
- You want CI to enforce "lockfile matches intent".
- You're managing the same skill set across multiple services.

For most teams, the explicit-prune procedure above is simpler and
sufficient.

### Demoting / forking a shared skill to project-only

Sometimes a project needs a variation of a shared skill that shouldn't go
upstream:

```bash
# Copy the current shared version into _skills/
mkdir -p _skills/code-review
cp -r .agents/skills/code-review/* _skills/code-review/

# Drop the shared lock entry
npx skills remove code-review

# Re-install as local
npx skills add ./_skills --skill '*' -a <agent> -y

# Now freely edit _skills/code-review/SKILL.md
```

## CI

Run the bootstrap commands as a CI step. They catch:

- **Shared skills deleted upstream** — `skills update` prints
  `✗ Failed to update <name>` and exits non-zero.
- **Broken local skills** — `skills add ./_skills` fails if any skill in
  `_skills/` has invalid frontmatter.
- **Lockfile drift** — if `skills-lock.json` references a ref that no
  longer exists, CI fails.

Example GitHub Actions step (substitute your own agent list):

```yaml
- name: Bootstrap skills
  run: |
    npx skills update -p -y
    npx skills add ./_skills --skill '*' -a claude-code -y
```

## Governance

### Name-space rules

- Pick a **prefix convention** for project-only skills to avoid collisions
  with shared ones. For example, if your service is `orders-api`, prefix
  local skills with `orders-` (`orders-migration-helper`,
  `orders-oncall-runbook`).
- Shared skills should use generic names (`code-review`, not
  `acme-code-review`) since they're meant to be reused.

### Ownership rules

- **Shared skills** are reviewed in the shared repo. Don't hand-edit
  `.agents/skills/<shared>/` in the consumer project — your change gets
  wiped on next `skills update -p`.
- **Project-only skills** are reviewed in the project repo. Don't hand-edit
  `.agents/skills/<local>/` either — your change gets wiped on next
  local-sync.
- The **single authoring location** per layer is:
  - Shared: the upstream repo at the pinned ref.
  - Project-only: `_skills/<name>/` (Variant B) or `<agent>/skills/<name>/`
    (Variant A).

### Review rules

- PRs that change `_skills/**` are **content changes** — review the skill
  body like documentation. The `description` field is especially
  high-impact (it controls dispatch).
- PRs that change `skills-lock.json` but not `_skills/**` are **shared-skill
  upgrades**. Review the upstream diff at the new pinned ref before
  merging.
- PRs that add a new shared-skill entry to `skills-lock.json` should
  include a link to the shared repo and the pinned ref.

## When the combined model doesn't fit

- **Per-developer skills** (not committed) — put them in the global scope
  with `-g`, or in a gitignored `_skills-personal/` directory with its own
  sync script.
- **Skills that need the shared repo's canonical content unmodified, but
  with project-specific `references/` files** — currently not supported
  cleanly. Options: (a) fork to project-only via demotion, then edit; or
  (b) keep the shared skill and add a sibling project-only skill that
  references it.
- **Offline / airgapped environments** — the CLI needs `git clone` access
  to refresh shared skills. Either vendor `.agents/skills/` into the repo
  (commit it, stop gitignoring), or host the shared repo on an internal
  git server the airgapped machines can reach.

## Quick reference

| Task | Command |
|---|---|
| Onboard a new machine | Run the bootstrap commands (shared update + local sync) |
| Add a shared skill | `npx skills add <repo>#<ref> --skill <name> -a <agent>` |
| Add a project-only skill | Create `_skills/<name>/SKILL.md`, then run local-sync |
| Edit a project-only skill | Edit file, then run local-sync |
| Update shared skills | `npx skills update -p -y` |
| Bump a shared pin | `npx skills add <repo>#<new-ref> --skill <name>` |
| Bump a pin + prune orphans | `skills remove` all old-pin skills, then `skills add` the ones that still exist at the new pin |
| Remove any skill | `npx skills remove <name>` (also delete `_skills/<name>/` if local) |
| Full reset (drift recovery) | Back up lockfile, `rm -rf .agents/skills <agent>/skills skills-lock.json`, re-add each skill explicitly |
| Promote local → shared | Copy to shared repo, `skills remove <name>` locally, `skills add <repo> --skill <name>` |
| Demote shared → local | Copy `.agents/skills/<name>/` to `_skills/<name>/`, `skills remove <name>`, run local-sync |
| List installed skills | `npx skills list` |

The "bootstrap commands" and "local-sync" referenced above are:

- **Bootstrap:** `npx skills update -p -y` followed by
  `npx skills add ./_skills --skill '*' -a <agent> -y`.
- **Local-sync:** `npx skills add ./_skills --skill '*' -a <agent> -y`.

Wire these into your task runner (`package.json`, `Makefile`, `justfile`,
shell alias, etc.) under whatever names suit your team.
