# Workflow

How we manage agent skills in our projects using the [`skills`](https://www.npmjs.com/package/skills)
CLI, across two complementary layers:

- **Shared skills** — reusable skills published in an internal git repo,
  consumed by many projects via `npx skills add`.
- **Project-only skills** — codebase-specific skills authored in-tree, not
  intended for reuse.

Both layers coexist in the same project without conflict. The CLI treats them
uniformly at install time but tracks them differently in the lockfile, so
updates and edits don't cross wires.

---

## Mental model

When you install any skill (shared or local), the CLI writes two things:

1. **Canonical copy** at `.agents/skills/<name>/` (project scope) or
   `~/.agents/skills/<name>/` (global scope).
2. **Symlink** into each target agent's skills directory — e.g.
   `.claude/skills/<name>/ → ../../.agents/skills/<name>/`.

The canonical copy is the single source of truth; every installed agent reads
through its symlink. If symlink creation fails (Windows without admin,
non-POSIX filesystems), the CLI falls back to a per-agent copy.

A project-scope lockfile at `./skills-lock.json` records what was installed
and from where. It's designed for version control:

- Sorted alphabetically (deterministic output).
- No timestamps (no incidental merge conflicts).
- Each entry carries `source`, `ref`, `sourceType`, and a content hash.

## CLI cheat-sheet

| Command | Purpose |
|---|---|
| `npx skills add <source>` | Install skills from a git repo or local path |
| `npx skills add <source> --skill <name>` | Install a specific skill from a source |
| `npx skills add <source>#<ref>` | Install pinned to a branch or tag |
| `npx skills remove <name>` | Remove skill + canonical copy + symlinks + lockfile entry |
| `npx skills update -p` | Refresh project-scope shared skills to their pinned refs |
| `npx skills update -p -y` | Same, non-interactive (CI-friendly) |
| `npx skills list` | List installed skills across detected agents |
| `npx skills init [name]` | Scaffold a new `SKILL.md` template |
| `npx skills find [query]` | Search public skills registry (skills.sh) |

Note: `-p` = project scope, `-g` = global scope, `-a <agent>` = target specific
agents, `-y` = skip confirmation prompts.

### Agent placeholders in this guide

Examples throughout use `<agent>` as a placeholder for whichever coding
agent(s) your project targets. The CLI supports many — Claude Code, Cursor,
Codex, Gemini CLI, Continue, Cline, Roo Code, Windsurf, and more. Run
`npx skills add --help` or browse the project README for the full list of
recognized `-a` values. You can repeat `-a` to fan out to several agents in
one invocation (e.g. `-a claude-code -a cursor`).

## What we commit

| Path | Commit? | Owner | Regenerated by |
|---|---|---|---|
| `skills-lock.json` | ✅ | CLI | `npx skills add` / `update` / `remove` |
| `_skills/**` | ✅ | You | hand-edited |
| `.agents/skills/**` | ❌ | CLI (generated) | bootstrap commands |
| `<agent>/skills/**` | ❌ | CLI (generated symlinks) | bootstrap commands |

(Concrete agent directories like `.claude/skills/`, `.cursor/skills/`,
`.codex/skills/`, etc. all fit the `<agent>/skills/` pattern.)

See [combined workflow](./03-combined-workflow.md#gitignore-template) for the
full `.gitignore` template and the bootstrap command sequence.
